# AISA 落地计划调整建议

日期 `2026-03-23`

这份计划的方向本身是对的。最值钱的判断有三个。第一，这个项目应该是 control plane，不是再造一个 worker。第二，第一阶段坚持文件系统优先，这让闭环能尽快跑起来。第三，人工 steer 必须是系统主路径的一部分，不是补丁。这三件事在 PRD、implementation blueprint 和当前代码里是一致的，所以主线不用推倒重来。

但下一步最需要收的不是功能清单，而是成功标准。现在文档写得比代码大很多，仓库里已经能跑通的是一个研究型 MVP。它可以创建 goal，固定生成三条 branch，用只读的 Codex worker 去分析，再靠规则打分生成 report。这个闭环证明了架子是通的，但还没有证明它是一个真正可用的 agent 工作台。真正的分水岭不在 dashboard 更漂亮，也不在 trigger 更多，而在于它能不能把一个真实任务稳定推进到可采纳结果。

所以计划上最该调整的一点，是先把第一批目标场景压得更窄。建议先只服务一种任务，最好是最贴近你们自己日常的那种，比如对一个本地代码仓库发起目标，拆出两到三个方向，一个方向负责结构理解，一个方向负责风险和验证，一个方向负责真实改动或可执行方案，然后允许人工在中途 steer，最后产出一个能继续开发的结论包。只要这条线稳定，后面的自动发现、cron、长期记忆才有意义。不然系统会先变成一个能展示很多状态，但很难真正收敛任务的总控台。

第二个需要调整的地方，是把 branch 类型分层。现在代码里的 worker adapter 只支持只读分析，这适合研究，不适合真正推进工程任务。后面如果要做 agent 工作台，至少要明确区分三类 branch。第一类是 research branch，只负责读和归纳。第二类是 execution branch，可以在隔离工作区里改代码、跑命令、产出 patch 和验证结果。第三类是 verification branch，只负责复核别的分支结论。没有这层区分，后面 planner、judge、预算控制都会混在一起，系统会越来越难管。

第三个要改的是 planner 的定位。现在的 planner 其实还是模板生成器，它固定吐出三条 branch，比较像演示数据，不像任务策略。下一阶段不该急着做复杂 DAG，而是先让 planner 至少能根据 goal 类型选 strategy。研究型 goal 就走 read-only 方案分叉，工程型 goal 就带一个 execution branch，排障型 goal 就优先证据链和复现。只要先做出两三种稳定策略，这个系统就会从 demo 跨到产品雏形。

第四个要改的是 judge。现在 judge 评分主要看 findings 数量、evidence 比例、next step 和 confidence，这能跑，但它评的是输出形状，不是任务结果。后面如果要让 keep、rerun、discard 这些动作有说服力，judge 必须更多依赖 goal 的 success criteria 和 branch 的验证产物。举个最直接的例子，如果一个 branch 声称方案可行，那它最好能附上跑过的命令、生成的 patch、失败日志或者测试结果。没有这些，系统保留的只是会写报告的分支，不一定是最有价值的分支。

第五个调整点是先补执行隔离，再谈多实例和复杂调度。只读研究阶段，直接对着 workspace root 跑问题不大。进入可写阶段后，必须尽早明确 branch 的工作副本、产物目录、回写方式和合并边界。不一定一开始就做成完整的 git automation，但至少要让每个 execution branch 有独立现场，能留下 patch、stdout、stderr、验证结果和最终建议动作。否则一旦多个分支开始写，同一个工作区会很快失控。

第六个调整点是把 dashboard 从总览优先改成 run page 优先。现在文档和 UI 都比较强调看板感，这对展示很友好，但真正使用时，用户大部分时间关心的是一个 goal 当前到了哪一步，哪条 branch 最可信，哪里卡住了，自己现在该不该介入。所以后面的界面重点不该先放在目标总数、分支总数这些概览，而该放在单个 run 的状态时间线、证据、最新 steer、生效边界和推荐动作。

如果按我来排下一阶段的顺序，我会这样收口。先定义一个真正要打穿的黄金路径，并把完成标准写死。这个完成标准最好不是“看起来跑通”，而是“一个真实 goal 能稳定创建、分叉、执行、人工介入、验证、收敛，并留下可复盘证据”。然后补 worker contract，把 read-only、execution、verification 三类 branch 的输入输出 schema 固定下来。再升级 planner 和 judge，让它们从模板和规则玩具，变成会看任务类型和验证结果的调度层。最后再补 SSE、trigger、长期记忆这些会放大系统规模的能力。

我会把 V1 的 Definition of Done 定得更硬一点。至少要满足这样一个标准。针对一个真实仓库任务，系统能自动生成合适的 branch 组合，其中至少一条 branch 能执行真实动作并留下完整证据，人工 steer 能在下一轮明确生效，judge 的结论和 evidence 能对得上，最终 report 不是摘要拼接，而是能支持下一步开发决策。做到这一步，这个项目才算从一个会跑的原型，变成一个值得持续投入的工作台。

整体判断还是偏乐观。现在最不该做的是继续往文档里加大词，也不是赶着做平台感很强的外围能力。最该做的是缩窄第一阶段的承诺，把能真正让你们自己每天愿意用起来的那条链路打实。只要这条链路成了，后面的扩展都会顺很多。
