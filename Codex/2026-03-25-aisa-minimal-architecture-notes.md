# AISA 最小架构认知整理

日期 `2026-03-25`

## 这份文档在回答什么

这份文档用来沉淀我们这几轮讨论后的一个核心判断。AISA 不能继续沿着功能名和模块名不断外长。真正优雅的状态，不是系统还可以继续加很多能力，而是删掉任何一个剩下的核心部分，系统都会失去成立条件。

我们希望借这份文档回答四个问题。AISA 最小不可再减的架构原语是什么。`planner`、`eval`、`state`、`context`、`report` 这些词在系统里各自应该处于什么位置。AISA 和 `karpathy/autoresearch` 的设计思想到底是同源还是偏离。以及最小可落地产品应该如何围绕这些认知来设计。

## 我们现在确认的核心判断

AISA 的本质不是一个新的 agent，也不是一个复杂的多模块平台。它更像一个围绕单个任务现场持续推进的控制面。这个控制面要做的事情很克制。它接住一个 goal，组织若干次尝试，允许人类中途介入，然后基于验证结果决定下一步是否继续、切换、回退还是结束。

所以架构的中心不应该是模块编排，而应该是任务闭环。我们真正关心的，不是系统里有没有 `planner`、`judge`、`context-manager`、`report-builder` 这些名字，而是一次 run 能不能被稳定创建、推进、验证和复盘。

## 最小不可再减的四个原语

### Run

`run` 是整个系统唯一的一等事实源。它代表一次真实任务现场。

一个 run 至少包含四类信息。第一类是契约，也就是这次任务要成为什么，通常包括 goal、success criteria、constraints、workspace root 和预算边界。第二类是历史，也就是任务推进过程中发生过什么，尤其是每一次 `attempt` 和每一次 `steer`。第三类是证据，也就是这个世界里真实留下了什么，比如 stdout、stderr、patch、命令回显、生成文件、测试结果和引用证据。第四类是当前判断，也就是系统此刻认为当前状态是什么、最优尝试是什么、下一步建议是什么。

这四部分里，真正的真相是契约、历史和证据。当前判断是从前三者中压缩出来的当前视图，不应该和前面三者并列为独立真相。

### Worker

`worker` 就是外部 agent，也就是 AISA 调出去执行某次尝试的执行体。

这里的“外部”不是说它必须运行在别的机器上，而是说 AISA 自己不负责完成具体推理和执行动作。AISA 只负责决定何时调用、给什么上下文、收回什么结果。至于真正去读代码、写总结、运行命令、生成 patch 的那个实体，就是 worker。当前这个角色就是 `codex exec`。

对于 AISA 来说，worker 最重要的不是品牌，而是契约。它必须接受一个上下文明确的任务，返回结构化结果，并留下可验证产物。

### Verifier

`verifier` 是系统里最容易被低估、但实际上最关键的原语。

它的职责不是生成内容，而是判断某次尝试到底有没有推进 goal。最简单的时候，它可能只是一个确定性的验证器，比如命令是否成功、测试是否通过、patch 是否生成。复杂的时候，它也可能包含规则判断、skill，甚至 agent review。但无论形式如何变化，它的角色都应该稳定，也就是系统中的判断真相。

这是 AISA 和很多“多 agent 系统”最容易拉开差距的地方。没有 verifier，系统只是不断地产生日志、摘要和分支。只有当 verifier 足够可信，系统才知道哪些尝试该保留，哪些该丢弃，哪些需要 rerun，哪些需要人工介入。

### Loop

`loop` 是系统里唯一真正主动的部分。它读取 run，调用 worker，等待 verifier 的结果，然后决定下一步如何继续。

loop 的职责并不神秘。它在每一轮只回答两个问题。刚才发生了什么。现在下一步最值得做什么。前一个问题主要依赖 verifier，后一个问题主要依赖 planning logic。两者合起来才是一轮真正的任务推进。

从这个角度看，系统最核心的不是模块之间互相调用，而是 loop 不断推动 run 演化。

## Planner 在哪

`planner` 很重要，但它不是一个必须先独立出来的顶层组件。

更合适的理解是，planner 属于 loop 里的 planning logic。它不是一个静态蓝图生成器，而是系统在当前时刻对“下一步最优行动”的判断。这个判断必然会随着 goal、最新证据、verifier 结论和人类 steer 持续变化。

这意味着 `plan` 不是事实源，而只是某一时刻的临时工作假设。它可以落地保存，方便 UI 展示和复盘，但它不应该被当成神圣蓝图。真正的事实源仍然是 run。

## Eval 在哪

`eval` 不是边角能力，也不是 report 的附属物。它应该被视为 verifier 的实现层。

我们已经确认，AISA 能否走远，核心不在于能不能再多起几个 worker，而在于能不能可靠地验证这次尝试是否真的推进了 goal。所以 eval 应该是一等公民。

但 eval 不应该先被等同于一个 agent。更准确的理解是，eval 是一个判断接口。简单场景可以是确定性验证，中间层可以是规则或启发式，复杂场景才使用 agent review 或 skill。这样系统既不会轻视 eval，也不会一开始就把整个验证体系做得过重。

在架构语言上，`judge` 不是一个独立宇宙，它只是 verifier 的一种实现方式。

## State 是什么

`state` 不是额外的一套真相。`state` 只是 run 在当前时刻的压缩视图。

比如 run 当前是 `running`、`waiting_steer` 还是 `completed`，当前是否有活跃 attempt，当前最佳结论是什么，当前下一步建议是什么，这些都属于 state。它们应该从 run 的历史和证据中派生，而不应该自己成为源事实。

所以更优雅的表达是这样。

`run` 是 source of truth。`state` 是 current view。`report` 是 human view。`plan` 是 next-step view。

真正不可混淆的只有第一件事，run 才是唯一事实源。

## Context 是什么

`context` 也不应该先长成一个独立大组件。

更好的做法是，所有 agent 都共享同一个 run 真相，只是在发起某次 attempt 时，从 run 中切出一份和这次任务相关的上下文，并冻结为这次 attempt 的 `context snapshot`。不同 agent 的区别，不是它们各自维护永久上下文，而是它们从同一个 run 中拿不同切片。

这样上下文不会漂，steer 也不会神秘丢失，复盘时也能明确知道某次 attempt 当时到底看到了什么。

## Report 是什么

`report` 不是任务现场本身，而是给人读的高层视图。

它的价值是帮助用户快速理解当前判断、最优结论、关键证据和下一步建议。但 report 不能替代原始证据，也不能替代 run。如果 report 和 run 之间出现冲突，应该以 run 为准。

所以 report 更适合被看成 `human view`，而不是单独的系统真相。

## 我们和 autoresearch 的关系

`karpathy/autoresearch` 对 AISA 的最大启发，不在于它用了单文件，也不在于它是 Python 脚本，而在于它只承认少数几个稳定原语。

在 `autoresearch` 里，真正不可替代的核心非常少。固定不让 agent 乱改的 verifier，也就是训练和评测真相。唯一的可变工作面，也就是 agent 每轮真正改的对象。人类写给 agent 的 program，也就是高层规则和运行方式。再加上一条非常简单的实验循环，修改、运行、看指标、保留或回退。

它的优雅，不在于功能少，而在于原语少。很多我们本能想做成独立模块的东西，在它那里其实都退回成 loop 内的一步动作或运行痕迹。

按这个标准照 AISA，我们之前确实有过偏离，而且偏离点主要有三个。第一，我们太早把派生视图当成了本体，把 state、context、report、events 往一级概念上抬。第二，我们太早把动词做成了名词，让 planning、judging、reporting 长成了平级大模块。第三，我们太早围绕多 agent 和模块编排搭系统，结果架构重心从任务闭环滑到了组件堆叠。

如果要重新贴近 `autoresearch` 的精神，AISA 就应该回到更少的原语、更强的 verifier、更薄的 loop、更厚的事实。

## AISA 的最小后端心智模型

到目前为止，我们最认可的一句话是：

`run = contract + attempts + evidence + current decision`

如果再加上系统运行视角，则可以进一步写成：

`AISA = run + worker + verifier + loop`

这四样东西足以支撑最小系统。其它今天看起来很重要的名词，大多都应该降回这四个原语的派生能力。

## 最小可落地产品应该怎么长

如果按这套认知落地，最小产品不应该先做全局大盘，而应该做一个围绕单个 run 的工作台。

产品中心是 run detail page，而不是全局 dashboard。用户打开系统后，最重要的页面应该能同时看到任务契约、attempt 时间线、执行证据、当前判断和 steer 入口。这样用户关心的核心问题可以被直接回答。任务要成为什么。系统刚才做了什么。留下了什么证据。系统现在准备怎么继续。人应该在哪里介入。

首页当然可以有 run 列表，但它只是入口，不是产品灵魂。

## 对后端落地的直接启发

如果继续沿着这份认知往工程设计推进，后端目录和数据结构也应该围绕 run 收拢，而不是继续平铺多套平行目录。

最自然的做法是，一次 run 就是一个任务现场目录，里面有任务契约、attempt 历史、steer 历史、原始产物和当前判断。state、report、plan 都只是从这个现场中派生出来的视图。

这也意味着，今天代码里的 `state-store` 在心智上更接近 `run-store`。真正存储的不是一堆抽象状态，而是任务现场本身。

## 这份认知对我们后续设计的约束

从现在开始，任何新增设计都应该先过一遍这几个问题。

这个东西是不是新的事实源。如果不是，它是不是应该只是 run 的派生视图。这个东西是不是新的顶层组件。如果不是，它是不是应该退回 loop 内的某个能力。这个能力是不是必须现在实体化。如果不是，是否可以先以内嵌策略存在。这个设计是不是在加强 verifier，还是只是增加系统表面复杂度。

如果一项设计不能清楚回答这些问题，那它大概率就是正在把系统变重，而不是变优雅。

## 当前阶段的最终共识

AISA 接下来最该做的，不是继续增加模块名，而是继续减少原语数量。架构上真正不可再减的核心应该是 run、worker、verifier、loop。planner 退回 loop，eval 归入 verifier，state/context/report 都降为 run 的派生视图。GUI 则围绕单个 run 的可观察和可干预展开。

只有这样，AISA 才有可能从一个会跑的原型，长成一个真正轻、稳、可验证的 agent 工作台。
